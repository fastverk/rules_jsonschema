# rules_jsonschema plugin contract

> Authoritative specification for what a codegen plugin must do.
> Plugins are invoked by per-language rules (`jsonschema_rust_library`,
> `jsonschema_go_library`, `jsonschema_starlark_codegen`, …) through
> registered toolchains.

## Process model

A plugin is any executable that conforms to:

```
INPUT
  stdin              the schema file contents (raw bytes, JSON)
  argv               --key=value pairs, repeated

OUTPUT
  stdout             the generated file content (raw bytes)
  stderr             human-readable diagnostics

EXIT
  0                  success — stdout is the generated file
  non-zero           failure — stderr explains why
```

Plugins must not write anything to stdout other than the final
generated file content. Diagnostics go to stderr unconditionally.

## Standard argv flags

The rule passes a fixed set of flags every plugin receives, regardless
of consumer wiring:

| Flag | Meaning |
|---|---|
| `--schema-name=NAME` | Basename of the source schema file, e.g. `compose-spec.json`. For error messages, codegen header comments, and (when relevant) deriving package / module names. |
| `--rule-name=NAME` | Name of the Bazel target invoking the plugin. Useful for picking output identifiers. |

A plugin **must accept** these flags. It may treat them as no-ops if
they don't apply.

## Consumer-supplied argv flags

User-facing rules forward consumer-defined flags through to the
plugin. Each rule defines its own set; example:

| Rule | Forwards |
|---|---|
| `jsonschema_rust_library` | (none today; rule wraps output in `rust_library`) |
| `jsonschema_go_library` | `--package=NAME` |
| `jsonschema_starlark_codegen` | `--kind=ID=POINTER:RULE_NAME:PROVIDER_NAME` (repeated); auto-kinds: `--kinds-pointer-base=POINTER`, `--kinds-pointer-suffix=SUFFIX`, `--kinds-key-filter=REGEX`, `--id-template=TPL`, `--rule-name-template=TPL`, `--provider-name-template=TPL` |

Plugins should treat unknown flags as a hard error — silently ignoring
them would make misconfigured options degrade the output without
warning.

## Error reporting

There are two distinct failure modes:

1. **Generation error** (the schema is valid, but the plugin can't
   handle some construct, or an option is invalid). Plugin writes a
   human-readable message to stderr and exits non-zero. Multi-line
   stderr is fine.

2. **Plugin bug** (the plugin crashed; a panic, segfault, library
   exception). Plugin process terminates abnormally; Bazel surfaces
   the non-zero exit and stderr.

Both cases fail the Bazel build identically. The exit code is the
authoritative success signal — stderr content alone is not a
failure.

Informational warnings on stderr **during a successful run** are
fine and will be surfaced in Bazel's build log. The discipline is
the inverse: stdout must contain *only* the final generated file
content, no progress chatter, never.

## Output stability

Plugins should produce **deterministic output** for the same input.
Specifically:

- Iterate over schema properties / definitions in **sorted order**
  (lexicographic by key). JSON's underlying maps are unordered, so a
  default `for (k, v) in obj` iteration is wrong here.
- Don't embed timestamps, hostnames, build IDs, or other host-state
  in the output.
- The full input path is not stable across consumers — use
  `--schema-name` (basename only) for header comments, never the path
  from `stdin`-as-file (there is no path; stdin is bytes).

A typical preamble:

```
// @generated by <plugin-name>. DO NOT EDIT.
//
// Source schema: <value of --schema-name>
```

## Multi-file output

A plugin emits **exactly one file's content** on stdout. Multi-file
output kinds (e.g. types vs validators, client vs server) are
expressed as separate Bazel rule invocations against separate
plugins — not as one plugin returning multiple files.

If a future plugin genuinely needs to emit many files at once, it
should be wrapped in a rule that uses `ctx.actions.declare_directory`
and downstream rules consume the tree artifact. The single-file
constraint is the default and what every existing rule expects.

## Concrete plugin skeletons

### Rust

```rust
use std::io::{self, Read, Write};
use serde_json::Value;

fn main() {
    let mut schema_bytes = Vec::new();
    io::stdin().read_to_end(&mut schema_bytes).expect("read stdin");
    let schema: Value = serde_json::from_slice(&schema_bytes).expect("parse schema");

    let schema_name = parse_flag("--schema-name").unwrap_or_default();
    // ... generate Rust source from `schema` ...

    io::stdout().write_all(generated.as_bytes()).expect("write stdout");
}
```

### Go

```go
package main

import (
    "encoding/json"
    "io"
    "os"
)

func main() {
    schemaBytes, err := io.ReadAll(os.Stdin)
    if err != nil { fail("read stdin: %v", err) }
    var schema map[string]any
    if err := json.Unmarshal(schemaBytes, &schema); err != nil {
        fail("parse schema: %v", err)
    }
    // ... generate Go source from `schema` ...
    os.Stdout.Write([]byte(generated))
}
```

### Python

```python
import json, sys
schema = json.load(sys.stdin.buffer)
# ... generate ...
sys.stdout.write(generated)
```

## Conformance testing

The `jsonschema_plugin_contract_test` rule (in
[`contract_test.bzl`](contract_test.bzl)) runs an in-repo Rust driver
against any plugin executable. The driver feeds the plugin a series
of inputs and asserts on stdout/stderr/exit behavior:

| Scenario | Asserts |
|---|---|
| `valid_minimal` | Plugin exits 0 + writes non-empty stdout for a well-formed minimal schema. |
| `malformed_input` | Plugin exits non-zero, writes to stderr, and **does not write to stdout** on garbage input. |
| `unknown_flag` | Plugin rejects an unknown `--key=value` flag (exit non-zero). |
| `determinism` | Two identical invocations produce byte-identical stdout. |

Plugin authors gate their toolchain registration with it:

```python
load("@rules_jsonschema//jsonschema:contract_test.bzl",
     "jsonschema_plugin_contract_test")

jsonschema_plugin_contract_test(
    name = "my_plugin_conforms",
    plugin = "//my:rust_codegen",
)
```

The default plugins shipped with rules_jsonschema (`schema_to_rust`,
`schema_to_starlark`, `schema_to_go`) each have a `*_conforms` test
target — adding a new plugin should follow that pattern.

## Versioning

Contract changes follow these rules:

- **Adding a standard argv flag** is breaking — plugins reject unknown
  flags. Coordinate across all known plugins before adding.
- **Changing stdin/stdout semantics** is breaking.
- **Per-plugin consumer-supplied flags** are owned by the rule that
  forwards them; changing those is the rule author's concern, not a
  contract issue.

The contract version is currently **v1**. Future versions will be
declared here and plugins are expected to support the latest.
